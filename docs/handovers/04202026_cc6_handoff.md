### Store purchase flow — rebuilt end-to-end

Nicole's original failure path: on `store.html`, the Buy Now button toggled open a single panel with BOTH the Zelle QR and the registration form. She scanned QR, paid via her bank app, closed the tab, and never filled the form. We got zero notification.

- **`store.html` refactored as a 2-step wizard** across all four buy flows (Beacon, Apex, Investigator Toolkit, generic checkout):
  - **Step 1** captures name/email/plan/school BEFORE the QR reveals. On submit, both ops `demo_leads` (INSERT with `REGISTER:` prefix) and Formspree fire in parallel. If *both* channels fail, the UI blocks progression and tells the buyer DON'T send Zelle, email support first.
  - **Step 2** reveals the QR, memo instructions, and a personalized confirmation ("Thanks Nicole — you're registered. We'll email your license key to nhill@magnoliaisd.org.").
  - Dropped the "Zelle confirmation #" field — it was post-payment data that didn't belong on pre-payment registration.
- **Server-side backstop:** added Supabase trigger `trg_notify_purchase_registration` on `demo_leads` INSERT. Fires Formspree via `pg_net` whenever a row's `referrer` starts with `REGISTER:` or `CLAIM:`. Verified end-to-end with smoke insert → email landed in inbox within ~15 seconds. Double coverage: browser-side + DB-side.

### The `/downloads/` Beacon trap

- `clearpath-site/downloads/Beacon-Counselor-Command-Center.html` was a 2 MB multi-chunk Vite build. Its JS module import (`./index-BNlXJZRD.js`) resolved to Cloudflare Pages' SPA fallback (the marketing homepage — 134 KB of HTML), so browsers tried to parse HTML as a JS module, init silently failed, endless "loading beacon" spinner.
- This — not a firewall — is what actually broke Nicole's first activation attempt. I spent 90 minutes chasing a pages.dev firewall theory before Kim's "AP's Apex worked on pages.dev" data point made me reconsider and her "loading beacon spinner" description locked in the real root cause.
- **Replaced** the 2 MB file with a 1.7 KB redirect stub: `<meta http-equiv="refresh">` + JS `location.replace()` + visible fallback link. Preserves `?key=...` query strings through the redirect. `<meta name="robots" content="noindex">` so search engines stop indexing the stub.

## What I got wrong today (write this down)

1. **Jumped to pages.dev firewall theory too fast.** Kim's phrasing "blocked by district" made me pivot off my empirically-verified original diagnosis (curl showed the `/downloads/` JS chunk served HTML). When the AP's Apex trial on pages.dev worked, I should have recalibrated immediately instead of doubling down. Signal: "loading beacon spinner" = Beacon shell rendered but JS didn't bootstrap. Firewall block = visible block page. Those are different failure modes.
2. **Premature URL swap.** Initially swapped 10 files `clearpath-apex.pages.dev` → `apex.clearpathedgroup.com` before checking VersionContext. `apex.clearpathedgroup.com` serves the Nationwide build, not Texas. Caught it before committing but only after writing the follow-up message to Kim. Rule for next time: read `VersionContext` / hostname routing logic before any URL swap on a product that might have multi-version deployment.
3. **Over-committed to "the pages.dev fix unblocks Nicole"** without evidence it was the right root cause. Cost Kim 30 minutes of waiting on the wrong theory before the real `/downloads/` trap was identified.

## Lessons worth keeping

- **A "loading" spinner is not a firewall block.** A block page is a block page. Distinguish before diagnosing.
- **Every SaaS product shipped on Cloudflare Pages needs a custom domain on day one,** not when a customer complains. Adding it pre-launch is one Cloudflare click; adding it reactively means sweeping code + emails + CORS + redeploying edge functions + updating auth config.
- **Feature-flag broken paths before Vera writes around them.** "Don't document the broken button" is worse than "there is no button."
- **Produce source-of-truth lists for delegation.** Vera doesn't need to verify features in the app; she needs a file saying yes/no/hidden. Same for any downstream agent.
