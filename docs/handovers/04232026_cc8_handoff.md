## What was done

### Beacon — critical fix to Clear Sample Data

`src/lib/seedSampleData.js`
- Root cause: `clearSampleData(counselorId)` did `SELECT * WHERE counselor_id=X` then `DELETE` for every row across ten tables. Anything the counselor had created after seeding was wiped — students, groups, sessions, referrals, time entries, notes.
- Fix: every row inserted by the seeder now carries `is_sample: true`. `clearSampleData` filters by that flag before deleting, scoped to the counselor.
- Legacy fallback: for installs that loaded sample data on the pre-fix build, the tag won't exist. When the tagged pass finds zero rows but the `beacon_sample_data` localStorage flag is still set, match the five known seed students (Marcus, Emma, Aiden, Sofia, Jayden) and their related groups/sessions/members/referrals by exact name. Counselor-scoped, safe.
- `SettingsPage.jsx` now shows the count of removed rows and warns that hand-entered students are never touched.

**Nicole's previously-lost students are unrecoverable.** Local IndexedDB is origin-scoped and had no backup. If she exported a JSON backup at any point, importing it would restore. User is texting her directly.

### Beacon — modal form-reset sweep

Same bug pattern was latent in 5 more Beacon modals. Applied the `useEffect(() => { if (open) { reset... } }, [open])` pattern to each:

- `StudentsPage.jsx` → `AddStudentModal` (Nicole's reported bug)
- `GroupsPage.jsx` → `NewGroupModal`
- `ReferralsPage.jsx` → `AddReferralModal` + `AcceptModal` (bonus — mode + selectedGroup persisted too)
- `SchedulePage.jsx` → `AddSessionModal`
- `GroupDetailPage.jsx` → `AddMemberModal`

Edit modals (`EditStudent`, `EditGoal`) already load from a prop correctly — not modified.

**Beacon commit:** `ff5161f` on `main`. Cloudflare Pages auto-deployed to `beacon.clearpathedgroup.com`.

### Cross-product audit — Apex, Toolkit, Waypoint

Spawned 3 parallel Explore subagents to check the other products for the same two bug patterns.

| Product | Destructive deletes | Modal form-reset |
|---------|--------|--------|
| Apex | 0 | 0 real (7 modals fit the pattern but parents use `{cond && <Modal/>}` conditional mount — remount resets state) |
| Investigator Toolkit | 0 | 0 (modals are vanilla JS, rebuild DOM via `innerHTML` on open) |
| Waypoint | 0 | 3 real — all in Meridian |

Why only Beacon had the destructive bug: it's the only product with a user-facing "Clear Sample Data" feature. Waypoint's `supabase/reset_sandbox.mjs` is scoped to one test district. Apex and Toolkit have no clear-demo feature.

### Waypoint — Meridian modal fixes

`src/pages/meridian/MeridianCAPTrackerPage.jsx`
- `LogFindingModal` — CAP finding#, description, legal citation, systemic/child due dates, tasks array — all reset on open. Previously only reset on successful save, so cancel + reopen left the form populated with the last finding's data.

`src/pages/meridian/MeridianStudentDetailPage.jsx`
- `EscalateModal` — urgency + notes both reset. Previously only notes reset on save; urgency stayed on 'urgent' (or whatever was last chosen) for subsequent escalations against different students.
- `LinkWaypointModal` — waypointId resets on open. **Catch worth flagging:** this modal takes a UUID linking a Meridian SPED student to a Waypoint discipline record. Without the reset, typing a UUID → cancelling → opening the modal against a different Meridian student would pre-populate the old UUID. One careless click and the wrong two students are linked. No reports, but it was live.

The shared `<Modal>` wrapper uses `if (!isOpen) return null` to hide its children, but the outer `LogFindingModal`/`EscalateModal`/`LinkWaypointModal` components stay mounted with `open=false`, so their `useState` persists. Exactly the Beacon pattern.

Shared `<Modal>` wrapper itself was not changed — changing it would force a refactor across ~40 call sites. Reset effects inside each modal is the minimal safe change.

**Waypoint commit:** `c49d177` on `main`. Cloudflare Pages auto-deployed to `waypoint.clearpathedgroup.com`.

### InviteUserModal — false positive

`UserManagementPage.jsx` → `InviteUserModal` fit the raw pattern, but the parent renders it with `{showInvite && <InviteUserModal ... />}` — React unmounts/remounts on toggle, so state resets naturally. Left unchanged.

## What didn't require a decision log entry

The tag-and-filter seed/clear pattern and the `useEffect([open])` modal-reset rule are both worth preserving as decisions, not just buried in a handoff. Two DECISIONS.md entries added:

1. **"Clear demo/sample" actions must delete only rows tagged by the seeder — never unconditionally delete all tenant rows.**
2. **Modals with an `open`/`isOpen` prop and hardcoded `useState` defaults must include a `useEffect(..., [open])` reset.**

Both are architecture-level rules that apply across products.

---

## Lessons worth keeping

- **Customer bug reports can expose latent patterns across the product suite.** Nicole reported two bugs in Beacon; the same form-reset pattern was present in 3 Waypoint Meridian modals and one of them (LinkWaypointModal) carried a real incorrect-link risk. Always audit siblings after confirming the original.
- **`if (!isOpen) return null` at the top of a modal does NOT remount it.** React only unmounts when the parent stops rendering the JSX entirely. To get fresh state, either (a) conditional-render at the parent (`{open && <Modal/>}`) or (b) reset state in a `useEffect([open])`. This was the subtle mistake in three separate products.
- **Local-mode IndexedDB has no server-side undo.** Beacon's FERPA-by-design local storage means a bad destructive operation is permanent. Worth thinking about whether Settings → Backup should nag the user more before the first destructive action (next-session candidate).
- **Subagent audits are worth the spend when a bug class might be codebase-wide.** Three parallel Explore agents checked three repos and returned actionable findings in under 2 minutes of wall-clock.
