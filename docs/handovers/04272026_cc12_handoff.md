## Decisions logged (2026-04-27, CC12)

Five new entries to DECISIONS.md:

1. **PDF attestation log via ops Supabase chosen over RFC-3161 / bonded TSA.** Constructing a TimeStampReq DER blob in browser JS is heavyweight (asn1.js / pkijs are both 50KB+); embedding the response as PAdES-LTV requires PDF tooling jsPDF doesn't ship. Instead Beacon POSTs `(counselor_id, document_kind, content_hash, generated_at)` to ops Supabase with anon-INSERT-only RLS and a server-default `now()`. Adversary's lawyer conceded this gives a colorable third-party witness; SCUTA's "self-witnessing" attack remains the wedge. RFC-3161 is on the district-readiness backlog for cloud-mode.

2. **Encrypted backup format: AES-GCM-256 with PBKDF2(license_key + email, 210k iters, SHA-256), magic-byte file header.** Format: `BCNBKP1` (7 bytes) + salt(16) + iv(12) + ciphertext. Restore auto-detects: magic = encrypted, `{` or `[` = legacy plaintext JSON. Backward-compat preserves all existing backups. Threat model: laptop-stolen / Downloads-leaked. NOT zero-knowledge (anyone with license + email can decrypt) — that is intentional for individual mode; district mode gets a district-managed KEK.

3. **Crisis occurred_at: "Now (locked)" by default, "Earlier than now" requires affirmative click.** Replaces the freeform `datetime-local` from CC11. Default mode pins `occurred_at` to current time on every render. Backdating requires deliberate UI action and produces an amber `RETROACTIVE ENTRY` banner on the PDF. Lawyer concedes this raises friction but is still cosmetic deterrence — the irreducible problem is that the counselor controls which version of any given record gets serialized.

4. **Turnstile site keys hardcoded in source as fallback.** Site keys are PUBLIC by design (Cloudflare embeds them client-side; only the SECRET key is server-only). After three failed Cloudflare Pages env-var retries, hardcoded `0x4AAAAAADEgqoGSVgO0m4aE` directly in `ReferralFormPage.jsx` as a fallback when `VITE_TURNSTILE_SITE_KEY` is unset. Env var still wins when set; useful for rotating keys without a code push. Zero security cost.

5. **Adversarial multi-voice audit before claiming defensive coverage.** Three voices (competitor / naysayer / hostile-CIO+lawyer) running in parallel with patch rounds in between produces drastically better results than internal review. Two specific patterns this session caught that internal review would not have: (a) the patch I shipped to fix the auto-backup risk introduced a NEW FERPA exposure (plaintext to Downloads, automatic = recurring); (b) the verify URL stamped on every PDF currently 404s, which the adversary called "self-inflicted credibility wound on every artifact." Both are **patches creating regressions**, and both were caught by the adversarial framing in ways static review or solo audit would have missed. Run this cadence whenever a release claims defensive / legal-grade properties.

## Lessons worth keeping

- **Patches create regressions, especially when claiming defensive coverage.** CC11+1 introduced two: the unencrypted Friday auto-backup (caught by round-2 adversary) and the 404'ing verify-attestation URL (caught by round-3 adversary). Both were caught only because of the multi-voice audit cadence; neither would have been caught by build/test/lint or a solo developer review. **Adversarial audit isn't optional for legal-defensibility claims.**
- **Marcia's 40% → 58% → 82% trajectory is a real signal.** The retention number moved 22 points in two patch rounds because the fixes hit the things she actually said had to be fixed (in the same words she used). When a naysayer's verdict moves that fast, the engineering is hitting the right targets.
- **SCUTA's "stop assuming they'll stay sloppy"** — the competitor moved from "single-counselor PWA priced on charm" (round 2) to "they're building a real product" (round 3) in one patch round. Keep the audit cadence going to verify subsequent rounds maintain that trajectory.
- **The verify URL pattern is a reusable defensive primitive.** A footer-stamped attestation receipt that points at a vendor-side verification page is a defensible chain-of-custody artifact. The pattern is reusable across Apex (observation PDFs), Waypoint (DAEP / SPED compliance PDFs), and Investigator Toolkit (case PDFs). Worth productizing once `verify-attestation.html` is built.
- **Cloudflare Pages env var path can fail silently.** `VITE_TURNSTILE_SITE_KEY` was saved in Production env, three retry-deploys ran on commit `e93bfce`, env var never made it into the bundle. Cause unknown — could be a Cloudflare propagation race, a Pages-vs-Workers env var precedence bug, or PEBKAC. Hardcoded fallback is the right resolution for public values; for actual secrets, would need a Worker proxy or a build-step override.
- **Three rounds of adversarial audit took ~25 minutes per round, ~75 minutes total of agent time, and produced material findings each round.** Strongly worth the cycles for any release that ships a "this is now defensible / compliant / FERPA-safe" claim.
