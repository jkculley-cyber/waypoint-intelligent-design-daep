### Migration 067 — corrective audit-trigger fix

First smoke run found `audit_log` rejected the trigger's INSERT: `column "actor_id" of relation "audit_log" does not exist`. **Production `audit_log` schema diverged from migration 035 source** — production has `(user_id NOT NULL FK, changes JSONB)` instead of `(actor_id, actor_role, old_values, new_values)`. This means `src/lib/audit.js` has been silently rejecting every Waypoint audit write since 035 landed (the `try { ... } catch {}` ate the errors).

067 rewrites `fn_navigator_audit_trigger` to use the production column shape. Also patched `src/lib/audit.js` and `src/hooks/useNavigator.js::useEntityAuditTrail` to write/read the same shape — Waypoint audit writes work for the first time since 035.

### Migration 068 — PL/pgSQL nested-IF fix

Second smoke run failed UPDATEs with `record "old" has no field "description"`. PL/pgSQL doesn't short-circuit `AND` inside IF *conditions* — `AND TG_TABLE_NAME = '...' AND OLD.description IS DISTINCT FROM NEW.description` evaluated the field reference regardless of operand order. Statement-level IF nesting *does* short-circuit, so 068 rewrote the audit trigger with nested IF blocks per table. Verified: 7/7 smoke tests pass.

## Decisions logged (2026-04-28, CC14)

Three new entries to DECISIONS.md:

1. **Navigator audit-log writes are fail-loud, not silent-swallow.** DB-level enforcement via trigger. Failure aborts the mutation transaction. Closes Chen's #1 finding ("audit failures must never block the user's primary action" was inverted: they must always block, otherwise the audit log is incomplete and a breach investigation has nothing to subpoena). Pattern reusable for any audited domain table — DB triggers writing to `audit_log` are inherently more reliable than app-layer `audit()` calls because they capture every mutation regardless of origin (UI, RPC, cron, import).

2. **PL/pgSQL trigger functions shared across tables of different shapes must use jsonb-based field access for table-specific fields, OR nested IF blocks.** Direct field references like `OLD.description` inside `AND TG_TABLE_NAME = 'navigator_referrals'` AND-conditions raise `record has no field` because PL/pgSQL doesn't short-circuit AND inside IF conditions. Statement-level IF nesting works (verified). Alternative: convert OLD/NEW to JSONB once and use `(jsonb ->> 'fieldname')` which returns NULL for missing fields. Documented in 068's header.

3. **Production `audit_log` has shape `(district_id, user_id NOT NULL FK→auth.users, changes JSONB, ip_address, user_agent)` — NOT the `(actor_id, actor_role, old_values, new_values)` shape declared in `supabase/migrations/035_audit_log.sql` source.** The migration source file is misleading; `audit.js` writes had been silently rejected by RLS/schema for the entire time 035 has been applied (since 2026-02-21). All app code using `audit_log` (audit.js, useEntityAuditTrail, the new Navigator triggers) now writes/reads the production shape. **General rule: source migration files are not ground truth for production schema — always probe production via `supabase.from('table').select('*').limit(0)` or a column-existence query before assuming.**

## Lessons worth keeping

- **Source migration files are not ground truth for production schema.** The `audit_log` divergence had been hiding for 8 weeks behind silent-swallow `try/catch`. When a trigger or app-layer write fails, audit telemetry needs to surface that — and surface it loudly. The fail-loud trigger pattern is a forcing function: any production schema drift now produces a visible HTTP 4xx instead of a silent void.

- **The adversarial multi-voice audit cadence applied to Navigator produced six convergent findings, not five.** Beacon's audits typically converged on 5; Navigator's converged on 6, including two that were blocking-on-multiple-axes (parent notice = TEC §37.009 + spoliation + Title VI evidence chain). Different products produce different convergent topologies — the audit framing is robust across product types.

- **PL/pgSQL gotcha: shared trigger functions across tables of different shapes need defensive field access.** Cost: 2 hours of smoke-test failure debugging. Three workarounds documented in 068's header. Reusable for any future trigger that fires on multiple tables (Meridian, Apex if it grows multi-table audit).

- **Smoke-testing under service-role + auth.uid()=null masks the real-user FK constraint.** Initial smoke runs failed test 3+6 with FK violation — that was a smoke-test artifact, not a production bug. Authenticating as the demo admin (real `auth.uid()`) made the tests representative. General rule: smoke tests for trigger-heavy code paths must run with realistic auth context, not service-role.

- **Marsha's 42% on Navigator vs Beacon's eventual 84% reframes the work timeline.** Navigator is materially behind Beacon in district-pilot readiness despite being labeled "live" in session-context.md. Round-2 audit will calibrate how much T3 work remains before "ship and call it 1.0."
