## Decisions logged (2026-04-30, CC16)

Five new entries in DECISIONS.md (AI Agent Team section):

1. **Constraint-tightening migrations must include a normalization/coercion step BEFORE VALIDATE.** The pattern: probe → backfill NULL → normalize known variants → coerce remaining unmappable values to a sentinel → ADD CONSTRAINT NOT VALID → VALIDATE. Migration 070 first failed VALIDATE because it backfilled NULLs but didn't address legacy non-NULL values that fail the new CHECK. Reusable pattern for any future constraint-tightening migration (any column where production has historical free-text data + we want to constrain it to an enum).

2. **Source migration files reaffirmed NOT ground truth — race vs race_ethnicity drift.** Migration 006 renamed `race_ethnicity` → `race`, but every later migration (066+) and all app code reference `race_ethnicity`. PostgREST queries silently failed, the hook's `|| 'not_specified'` fallback hid it, and Disproportionality by-race has been a no-op since shipped. Third confirmation in three weeks (CC14 audit_log shape, CC15 audit.js entity_id, CC16 race column). Reaffirms: probe production schema via PostgREST `.select('col').limit(0)` before assuming any column exists. Adds: when drift is detected, the fix migration must be defensive (handle all states — old name only, new name only, both, neither).

3. **Schema-drift CI tripwire pattern: probe column existence on app-load-bearing tables.** Not a full schema diff. A focused tripwire: `scripts/check-schema-drift.mjs` calls `.select(col1,col2,...).limit(0)` on each tracked table. Missing or renamed columns surface as PostgREST errors, fail the build. CI runs on push + PR + manual via `.github/workflows/schema-drift.yml`. Cost: <2 hrs to ship; benefit: would have caught the audit_log shape divergence and the race_ethnicity column rename within a day. Reusable pattern for any application reading from a managed-schema database.

4. **Sentinel UUID seeded as real auth.users row vs dropping the FK.** Two design options for closing Chen R2-1: (a) seed the all-zeros UUID as a real `auth.users` row with a non-loginable password, (b) drop the `audit_log_user_id_fkey` FK constraint. Chose (a) because the FK still serves as a safety check for non-sentinel user_id values; the sentinel insert is one-time + idempotent (`ON CONFLICT DO NOTHING`). The sentinel row's `email = 'system+sentinel@waypoint.internal'` and `encrypted_password = '$AUDIT_SENTINEL_NO_LOGIN$'` (non-bcrypt) so authentication against it always fails. Reusable pattern for any FK-anchored sentinel.

5. **JSONB integrity CHECK constraints: required-keys + monotonic time-ordering for any history array column.** Migration 072 applied two CHECK functions (`fn_validate_reason_history`, `fn_validate_description_history`) that enforce shape (each row has the expected keys) + temporal order (`changed_at[i] <= changed_at[i+1]`) + sanity (`changed_at <= now() + 1min`). Raises friction floor on direct-SQL JSONB mutation attacks (Reyes #1) without requiring a separate immutable audit_edit_log table. NOT VALID/VALIDATE pattern grandfathers legitimate existing data (e.g., the CC15 walkthrough seed) if it fails — surface in a follow-up audit, don't reject the migration. Reusable for any future history-array JSONB column.

## Lessons worth keeping

- **Three drift discoveries in three weeks.** CC14 audit_log shape, CC15 audit.js entity_id, CC16 race column rename. The pattern of "source-controlled migration applied → out-of-band ALTER → drift" has happened at least three times. The schema-drift CI tripwire we shipped today is the first preventive measure; a `pg_dump` audit is still the missing comprehensive sweep.

- **Constraint-tightening migrations need normalization, not just backfill.** I shipped migration 070 with NULL backfill but no legacy-variant normalization, betting the existing data was already canonical. It wasn't (PEIMS codes + natural-language variants). Same trap will hit any future migration that takes a free-text column and constrains it to an enum. Pattern is now formalized in DECISIONS.md.

- **Audit cadence convergence: round 3 surfaced 4 convergent + 3 single-voice findings.** Round 1 had 6 convergent. Round 2 had 6 convergent. Round 3 had 4. The curve is flattening in the right direction — Marsha + my recommendation to stop at round 3 (no round 4) is data-driven, not vibes.
