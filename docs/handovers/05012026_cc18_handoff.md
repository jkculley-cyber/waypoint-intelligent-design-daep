### 2. R4-4 — Behavioral operands in STEP 0 (commit `d44d936`, ~30 min)

Closes Wallace R3 NEW #1 ELEVATED: the principle-based bias guardrail from R3-1 (CC17) closed the enumerated-list attack but left coaching output free to paraphrase actions/counts/durations into adjective-shaped fluff ("called on multiple students" instead of "called on 5 students"; "appropriate wait time" instead of "8 seconds of silence").

**Both `generate-coaching-draft` + `ib-coaching-draft` system prompts:** added a PROVENANCE OPERANDS block requiring four operand forms:
- ACTIONS (verb + object) — "called on Marcus", not "engaged students"
- COUNTS (integers) — "5 students" / "3 of 24", never "many" / "several" / "engaged"
- DURATIONS (seconds/minutes) — "8 seconds of wait time", never "appropriate" / "good" / "lengthy"
- QUOTES (exact words from transcript in quotation marks)

Explicit instruction: if the source transcript contains an action/count/duration/quote, preserve it verbatim. Reject paraphrase that strips counts/timing/exact-words. If source doesn't contain operand-shaped detail, do NOT invent — say so explicitly OR fall to "Not Observable" rather than fabricate.

Three example transformations in `generate-coaching-draft` (one new contrasting "asked good questions and waited a while" → real operand-shaped output) and a compressed version in `ib-coaching-draft` matching that function's flatter system-prompt structure.

Watched fields differ per table: observations gets ai_draft + principal_edits + final_content + pre_conference + post_conference + ttess_dimension_notes + ttess_ratings + area_of_reinforcement + area_of_refinement + structured_feedback + ib_alignment + status + sent_to_email; ib_observations gets voice_transcript + ai_summary + coaching_notes + practice_tags

## Decisions logged (2026-05-01, CC18)

One new entry in DECISIONS.md (Product & Features section):

**No forced login for the signing/receiving party in any Clear Path product.** When the documented or receiving party in a product needs to sign, acknowledge, or respond to a record — default to a magic-link / token flow, never force account creation or login. Applies to: Apex teachers signing post-conference reflections (R4-2), Beacon parents acknowledging crisis notifications, Waypoint parents signing DAEP placements, Origins family-portal participants, future suite products. Bobby (real Apex principal customer) churned earlier in Apex's life specifically over teacher-facing login friction — the receiving party has zero motivation to maintain an account in a vendor's system they didn't choose; the buyer (principal/counselor/AP) has the relationship, the receiving party has a transactional obligation. Magic-link safeguard pattern: single-use token, ~14-day expiry, capture `signed_at + signed_via='email_link' + signed_ip` so provenance is verifiable without account-anchored auth. Full login UI is explicitly Tier 2/3, only justified when a district contract REQUIRES account-anchored auth. "More defensible against spoofing" arguments lose to "the feature has to ship and get used" — magic-link record + IP capture is enough provenance to defend; a missing signature because the teacher never logged in is indefensible.

## Lessons worth keeping

- **Bobby decision: no forced login for receiving parties.** Cross-product rule that should govern future Beacon parent flows, Waypoint parent acknowledgments, Origins student/family flows. Saved as feedback memory + DECISIONS entry.

- **Smoke-test ping-pong is a real anti-pattern.** When a deploy + verify loop has 3+ rounds with partial confirmations on each round, the user perceives it as wheel-spinning even when each individual round is technically "doing diagnostics correctly." First-round verification should be comprehensive (full response body, not just status code; correlated DB query, not just function output). Save the iterative-narrowing pattern for cases where the cost of being wrong is high. Saved as feedback memory.

- **Migration policy idempotency.** `CREATE POLICY` has no `IF NOT EXISTS`. Default pattern for any policy in a future migration: `DROP POLICY IF EXISTS "name" ON table; CREATE POLICY "name" ...`. Cheap insurance against re-paste errors in SQL Editor (which can't be cleanly differentiated from "did the migration actually fail" vs. "did it actually succeed but I don't know"). Worth adding to the migration template.

- **R4-1 trigger pattern (BEFORE UPDATE + to_jsonb shape-tolerance + per-field comparison loop) is reusable.** The same shape works for any "audit every edit to a known set of columns on this table" requirement across products (Beacon session-note edits if cloud mode lands; Waypoint incident-row edits if a SEVERE finding ever surfaces there). Header documentation explains the CC14 lesson it embodies.
