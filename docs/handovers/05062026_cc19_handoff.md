## Decisions logged (2026-05-06, CC19)

One new entry, **Architecture / Data Model** section:

**Calendar entries that aren't counseling sessions live in their own `schedule_events` table/store, not in `sessions` with a relaxed CHECK constraint.** Beacon's pilot feedback included "schedule non-counseling things — meetings, duty, etc." Two design options: (a) extend `sessions` with `session_type IN ('individual','group','event')` and a nullable `event_type` discriminator + `title` field; (b) add a separate `schedule_events` store. Chose (b) because: (1) `sessions` carries downstream semantics that don't apply to events — `autoLogTime` only fires for individual/group Completed, SOAP-note flow is counseling-specific, PEIMS reporting if it ships will be counseling-specific, status enum (Scheduled/Completed/Cancelled/Make-up Needed) doesn't map cleanly to events; (2) every consumer of `sessions` (SessionsPage filter, dashboard counts, weekly bar chart, student detail page) would need to learn about a new type and filter it out — that's CC8-shape failure-mode propagation; (3) events have schema headroom they shouldn't have to share — `title TEXT NOT NULL`, `event_type` enum with its own evolution path, no need for `student_id` / `group_id` / `objectives_covered` / `lesson_ids`. Cost: one extra IndexedDB store + ~15 lines of duplicated query plumbing in the load functions. Pattern is reusable: when a calendar feature shares date/time/title with sessions but has different downstream semantics (no auto-log, no instructional reporting, no clinical workflow), it gets its own store. Same applies to future Apex calendar entries that aren't observations, future Waypoint calendar entries that aren't placements/reviews, etc.

## Lessons worth keeping

- **Real-pilot-customer feedback ships same-session when the work is in a local-mode product.** Beacon Cloudflare Pages auto-deploys on push to `origin/main`; IndexedDB schema bumps are zero-friction (`onupgradeneeded` handles the upgrade in the background); cloud-mode migrations can be staged for "when a district lands" without blocking the local-mode ship. Nicole texted at hour zero, all six asks were live by hour two. Pattern is NOT reusable for Waypoint/Apex/Navigator (cloud-only, require migration apply step + service worker bust check). It IS reusable for Investigator Toolkit (the other local-mode product) and any future local-mode-first product.
- **When asked "is X searchable?" — the honest answer plus one good follow-up question is more useful than just yes/no.** Nicole asked "is schedule searchable?" Answer: no, the calendar grid isn't, but Sessions page text-search exists. The follow-up I offered ("most counselors who ask for schedule search really want an agenda/list view") got a one-word "yes" and unlocked a feature that materially improves the product. Pattern: if a customer asks "does X exist?" and the answer is no, propose what they'd ACTUALLY want next, not just confirm the gap.
- **Status enum visible on calendars needs both color/style AND a glyph, not one or the other.** Color alone fails on small cells (10×10 monthly dots) and for colorblind users; glyph alone fails when the cell is too narrow to render the glyph. The combination (icon + boxShadow inset / opacity / dashed border) works at any size — small cells get the style cue, larger cells get both. Same pattern will apply to Apex observation status, Navigator placement status, Meridian timeline status when those land in calendar-shaped views.
- **`schedule_events` separate from `sessions` was the right call** (logged as a decision). The CC8 lesson — "every consumer of a multi-shape table is a future bug" — held. Adding event_type to sessions would have polluted SessionsPage filtering, dashboard counts, weekly bar chart, autoLogTime semantics, and PEIMS reporting if it ships. The separate store cost ~15 lines of duplicate query plumbing in load functions and zero downstream complexity.
