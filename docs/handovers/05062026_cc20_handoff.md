## Decisions logged (2026-05-06, CC20)

**Pricing & Go-to-Market section:**

**Apex API costs are owner-borne until token budget per tier ships.** Every principal's `generate-coaching-draft` / `ib-coaching-draft` / `rewrite-narrative` / `transcribe-observation` call hits a single shared `ANTHROPIC_API_KEY` (and OpenAI key) on the Supabase Edge Function project. Per-observation cost ~$0.05-0.50 (Whisper transcription + Claude coaching draft, possibly 2× with rewrite). Manageable at $99/mo retail (~5-15% of revenue at typical 30 obs/mo); 25-80% of revenue at $19/mo intro pricing. **Three structural options on the radar** (must pick one before Apex starts converting at scale): (a) **Token budget per tier** — track usage in `principals` table, hard-cap per month (50 obs Standard, 200 Pro). Simplest. (b) **BYO API key** — each principal pastes their own Anthropic key in Settings. Offloads cost but defeats easy-adoption. (c) **Pass-through metered billing** via Stripe usage records. Right answer at scale, real engineering lift. **Until one ships: every Apex pilot/trial/paying user is owner-borne cost, period.** **Lesson:** cost-per-customer math should be sized at product launch, not at first credit-exhaustion event. The architectural assumption ("we'll pay for the API at first") becomes increasingly indefensible as conversions ramp; pricing the product without the COGS line item is how SaaS founders get surprised by "100% of revenue is OpenAI invoices" stories.

## Lessons worth keeping

- **Cost-per-customer math should be sized at product launch.** Tonight's credit-exhaustion event is the kind of thing that would have been a 30-minute conversation at Apex launch and a multi-day "what do we do about this" fire-drill at first scale. The architectural assumption "we'll pay for the API at first" gets indefensible as conversions ramp. New DECISIONS.md entry tonight makes this concrete; ideally surfaces in every future product launch checklist before the Stripe page goes live. (User's specific feedback: disappointed this wasn't surfaced earlier — fair criticism.)

- **Supabase Edge Function secrets sometimes need a function redeploy to propagate.** Tonight's Anthropic key troubleshooting circled through "wrong project / wrong name / typo / stale runtime" before redeploy unblocked it. This isn't documented prominently in Supabase UI; counts as institutional knowledge. Worth a memory entry so the next time the symptom appears (function reads env var, gets undefined despite secret being set), redeploy is the first move not the fifth.

- **Migration apply timing got compressed tonight in a good way.** Beacon 004+005 + Apex 019+020 all applied within hours of code being staged, not days as in past sessions. Pattern: when migrations are idempotent and the user has SQL Editor access, "stage + apply same session" is faster than "stage + defer to next session." Worth defaulting to this when possible.

- **Don't over-rotate on smoke-test loops, but DO ask for the literal response body when an HTTP error message is generic.** R4-3 deploy verification has been "function returns N (template / 500 / something)" three sessions running because we kept guessing what the body says instead of asking for it directly. The Supabase dashboard Test panel always shows the body below the status — that's the actual diagnostic, not "did the smoke pass."

- **Closing-process compression worked well.** This session ran tight: ship → smoke → diagnose → ship next → smoke → close. The friction came from credit exhaustion (genuinely external blocker), not from process. Keep this cadence when there's clear queue ordering.
