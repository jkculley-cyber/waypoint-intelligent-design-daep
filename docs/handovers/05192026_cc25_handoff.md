## Lessons worth keeping

- **The Supabase deploy pipeline has two independent layers — and conflating them costs an entire user feedback round.** User updated the Edge Function via Dashboard and reasonably believed "demo flow fixed." But the Edge Function only controls the email backend; the *buttons themselves* are static HTML in Cloudflare Pages. Buttons stayed broken until PR #1 merged to `main` and Cloudflare redeployed. Next session, when a multi-layer fix is shipped, lead with the deploy-state diagram before the user starts testing. Pattern is recoverable but it ate ~2 user-frustration messages before it surfaced. Generalizes to every Supabase+Cloudflare combo: Edge Function deploy ≠ frontend deploy, ever.

- **"Fix it" means merge it.** User said "Fix it" after I asked whether to open the PR — I opened it but didn't merge. They saw the live site still broken, said "you are wasting my time." Should have read the brevity of "Fix it" as "make the working state live" not "create the artifact that could become the working state." PR-as-artifact thinking is wrong when the user owns the deploy gate and is testing the live site immediately. Heuristic: when the user has authorized the change and is actively waiting on the result, take the action all the way to "in front of users" not just "in code review."

- **Supabase CLI `functions deploy` from a directory that doesn't contain `supabase/functions/<name>/index.ts` produces a misleading error chain on Windows.** `WARN: failed to read file: open supabase\functions\send-notification\index.ts` → `unexpected deploy status 400: Entrypoint path does not exist` → followed by `Invalid Function name. Must start with at least one letter...` (which is misleading; the name is valid, the CLI is just tripping on `npx supabase`-version-specific behavior). When the CLI errors out for a user on Windows, dashboard upload is the unblocking move — drag the file into the Edge Functions UI and skip the CLI entirely.

- **Marketing-site demo creds must be the auto-resetting sandbox, not the curated internal demo district.** Lone Star ISD demo accounts (`admin@lonestar-isd.org` etc.) are seeded with specific incident/compliance state that supports guided walkthroughs and could be disturbed by anonymous-public traffic. Explorer ISD (`supabase/seed_sandbox.mjs`) is purpose-built — periodic reset, manufactured data only, public-traffic-safe. Catching `ap@lincoln-hs.demo` (an account that never existed) on Navigator's success page was a separate bug but in the same family: any account name in marketing copy must trace back to a real seeded account, and the only safe one to expose publicly is Explorer ISD. Worth a periodic grep across `clearpath-site/` for hardcoded email strings.

- **One CTA per intent. Two buttons going to the same place is just doubled friction with no clarity benefit.** The Navigator hero had "Request Free Demo" (form → creds + email) and "View Live Demo" (login wall, no creds) — both ended at `waypoint.clearpathedgroup.com`. Removing the duplicate was a one-line change with zero downside. Same pattern probably exists across the rest of the marketing surface area; worth a periodic CTA-duplication audit.
