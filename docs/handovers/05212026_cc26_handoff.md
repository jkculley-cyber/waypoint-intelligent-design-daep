## Lessons worth keeping

- **Source migrations are not ground truth, fourth time this quarter.** CC14 found audit_log shape divergence, CC15 found audit.js entity_id NOT NULL gap, CC16 found students.race column rename, and **CC26 found audit sentinel rows missing from production despite migration 071 supposedly having seeded them.** The pattern: schema-drift CI catches **columns**, but row-level invariants (sentinel rows, baseline config, required seed data) also drift, and the cost of discovering this in production is a multi-round seed-debug cycle. The fix: extend schema-drift CI to probe required-row existence too â€” sentinel UUID present in profiles, default district/campus seed rows present, any rows the audit FKs depend on. ~1hr addition.

- **`auth.admin.listUsers()` is not a reliable primitive past a user pool threshold.** Returns 500 "Database error finding users" server-side on the Waypoint project. Affects every seed script that uses it for an "already exists" check. Two scripts fixed this session via two different fallback patterns: (a) profile-table email lookup (for seeds where the user already has a profile), (b) `signInWithPassword` to recover UUID (for seeds where the password is known and the profile may not exist). Pattern reusable for any future seed/maintenance script.

- **Mid-execution patching is the signal that a seed script has drifted behind production.** This session's `seed_navigator_campus_demo.mjs` hit FIVE separate latent bugs â€” listUsers, race column rename, race CHECK, parent_notified_at trigger, parent_notified_method NOT NULL â€” each from a migration that landed AFTER the seed was last validated. The seed had simply never been run in production. **Rule:** any "never been run in production" seed needs a production validation pass before being relied upon in execution-walkthrough mode. The latency between "we wrote the seed" and "we ran the seed" can hide every migration that landed in between. Worth running ALL seeds quarterly against production as a cross-check that schema-drift CI alone doesn't catch.

- **Multi-product PWAs need product-scoped `from`-path validation in LoginPage.** When a district's products array changes (this session's exact case for Explorer admin), browser history can carry a URL for a product the user no longer has access to. LoginPage previously followed `from` blindly as long as it wasn't `/waypoint-admin` â€” that ate the explore@ admin's first post-merge login. Pattern applies to any multi-product app: validate route prefix against `hasProduct()` before following, otherwise users hit confusing "not included in your plan" walls instead of their actual landing page. Beacon (when district-multi-tier ships), Apex (if it gains role tiers), future Clear Path products all need this validation.

- **LocalStorage dismiss keys must be scoped to the entity they dismiss, not global.** The `sandbox_cross_sell_dismissed` key was global per-origin, so dismissing on one sandbox suppressed the banner on the other. Per-district key fix (`sandbox_cross_sell_dismissed_<district.id>`) shipped. Same trap applies to any future "dismiss this banner / nudge / hint" UI â€” must be scoped to the context (district, user, principal, counselor) it's dismissing, not the browser. Beacon morning brief dismisses if added, Apex coaching nudge dismisses, future Clear Path cross-sell surfaces.

- **The latent collision in `seed_navigator_campus_demo.mjs` had already been triggered once in production history.** Evidence: Explorer ISD had `ap@lincoln-hs.demo` + `counselor@lincoln-hs.demo` profiles in it (which we cleaned up this session). That means at some point between CC11 (when the seed was written) and CC26 (now), someone ran the seed against the Explorer ISD UUID, deleted Explorer ISD's profiles, and reseeded with Lincoln HS data â€” but Explorer ISD's UUID, so the data was attributed to the wrong district. The CC25 finding of `ap@lincoln-hs.demo` referenced in marketing copy was downstream of this. **Pattern:** any seed script that does `DELETE FROM districts WHERE id = HARDCODED_UUID` is a high-blast-radius operation that needs an explicit "are you sure" guard or an environment-scoped flag (`DEMO_ENV=staging`). The cost of a misfire is total district wipe.
