The leaked keys are already revoked, so the values sitting in history are inert. A `git filter-repo` + force-push rewrites every SHA and breaks all existing clones/PRs for near-zero security gain. A scrubbed mirror was produced at `/tmp/wp-scrub.git` (rewritten main `f796b9b`, 0 JWT headers, 40 `REMOVED` markers) but **NOT pushed**. Pre-scrub recovery anchor = `c886d2f`. The correct disposition of the GitHub alert is to **dismiss it as "revoked,"** not scrub.

## Lessons worth keeping

1. **Disabling "legacy API keys" is the ONLY thing that revokes a leaked legacy service_role.** Creating/rotating new keys does nothing to the old value. The dashboard toggle kills legacy anon + service_role *together* — so before flipping it, confirm every server env (Functions `WAYPOINT_ANON_KEY`, proxies) is already on the new keys, or both proxies break at once.
2. **Vite content-hash is a deploy truth-teller.** An unchanged output bundle hash after a "successful" deploy means the build inputs (env vars) never changed. Used twice this session to catch a build var that hadn't actually been swapped.
3. **The two repos deploy differently.** Waypoint = GitHub Actions (`VITE_*` from GitHub repo secrets). clearpath-apex = Cloudflare native git integration (`VITE_*` from Cloudflare dashboard build vars). The publishable-key swap location differs per repo — don't assume one mirrors the other.
4. **Skip the scrub when keys are already revoked.** Reserve the destructive force-push rewrite for *live* secrets or repos about to go public. Otherwise dismiss the alert as revoked and spend the effort on forward-looking controls (gitleaks) instead.
