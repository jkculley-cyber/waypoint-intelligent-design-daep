## Headline

User reported (with screenshots) that **Apex — currently suspended ("Coming Soon") — was rendering a green "LIVE NOW" corner badge** on both its cards on `clearpathedgroup.com`: the hero-carousel slide and the `#apex-pricing` card at the bottom. The "LIVE NOW" badge sat directly above/beside the orange "Coming Soon" badge on the same card — a clear contradiction.

**Fixed, shipped, deployed, and verified live.**

---

## Root cause

A **global** CSS rule injects the badge as a pseudo-element:

```css
/* clearpath-site/index.html:224 */
.path-card::before { content: 'LIVE NOW'; /* green gradient, top-left corner */ }
```

It stamps "LIVE NOW" on **every** `.path-card`. Both Apex cards are `.path-card`s, so they inherited it. The HTML *text* already said "Coming Soon" everywhere (carousel, role-picker, pricing section, the `/apex` page) and the source was committed clean since `b6390dd` — which is exactly why my first two checks (a `curl | grep` of the live site and a source read) **missed it**: the badge is a CSS pseudo-element, not page text.

## Fix

- Added a modifier rule: `.path-card.no-live-badge::before { content: none; }`
- Applied `no-live-badge` to the **two Apex cards only**:
  - Hero carousel slide — `clearpath-site/index.html:828`
  - Pricing card `#apex-pricing` — `clearpath-site/index.html:1249`
- Live products (Waypoint / Navigator / Beacon) keep their "LIVE NOW" badge untouched.

The role-picker Apex card (`index.html:947`) is a plain styled `<a>`, not a `.path-card`, so it was never affected.

## Decision logged (DECISIONS.md — Architecture & Technology)

A product's "live"/status badge can be a global CSS pseudo-element invisible to text/HTML/curl-grep checks. When suspending a product, also suppress any such `::before`/`::after` badge, and verify the **rendered** state via `getComputedStyle(el,'::before').content` — not page text.

## Lessons worth keeping

- **Verify the rendered output, not just the source text — pseudo-elements don't show up in `curl | grep` or a source read.** Two passes reported "everything says Coming Soon" and both were right about the HTML and wrong about what rendered. The badge only surfaced once the user sent a screenshot and I grepped the CSS for `LIVE NOW`. For any "it shows X but shouldn't" report on a styled UI, check computed styles / pseudo-elements early.
- A global decorative rule keyed on a structural class (`.path-card`) silently applies to new/edge cases (a suspended product reusing the same card). Status badges should be opt-in per card, or at least have a documented opt-out — the `no-live-badge` modifier is that opt-out now.
