## Root cause (verified by static trace, not guessed)

A normal trial counselor is created by `createLocalCounselor` (`src/contexts/AuthContext.jsx`) with `subscription_status: 'trial'` + `trial_started_at` and **no license key** — a key is only stored *after purchase*. The launch auto-backup in `AppShell.jsx` is gated on `if (!licenseKey || !counselor.email)`, so for the **entire trial it was silently skipped**. The `beacon_backup_needs_license` flag it set "to prompt the UI" was **written at AppShell.jsx:166 and read nowhere** (grep-confirmed across `src/`). Net effect: a brand-new counselor — the cohort most likely entering first real data on an eval/shared device — got **zero automatic backups and zero notice**, with the staleness banner additionally suppressed the first 48h (`isNewUser`) and only firing at ≥7 days. Off-device protection only switched on **after they'd already paid** — the trust spine inverted.

Confirmed deterministic (gated on a `localStorage` read + `subscription_status`), so no runtime ambiguity. Browser automation was **not** reachable this session (preview MCP is locked to the Waypoint project root and can't attach to the sibling `clearpath-beacon` repo; no Chrome connected to the Chrome MCP; computer-use disconnected) — the finding stands on the static trace + the license/trial model.

## Fix

**Gap A — surface the no-auto-backup state** (`src/components/layout/AppShell.jsx`):
- Removed the dead `beacon_backup_needs_license` flag.
- `AppShell` now computes **why** auto-backup isn't running — `autoBackupBlocked`: `'trial'` (no license → can't encrypt) or `'no_folder'` (licensed but no off-device folder) — and renders a real, actionable banner once the counselor has data worth protecting (`hasData` from the existing sidebar counts). Closes the trial blind window the staleness banner missed.
- Banner variants: trial-no-folder → "Automatic backups turn on when you activate… set a backup folder + back up now" + **Set up backup folder** / **Back Up Now**; trial-with-folder → "backs up when you click — auto starts at activation"; licensed-no-folder (Chromium) → "pick a backup folder so Beacon backs up automatically".

**Gap B — drive folder setup + stop the staleness clock lying**:
- Onboarding step 5 (`src/components/OnboardingChecklist.jsx`) changed from "Set up a backup reminder" (complete when *any* backup ever happened) → **"Set up an off-device backup folder"**, complete when a folder is actually **configured** (`getBackupFolderName`). Firefox/Safari (no folder API) fall back to "a completed manual backup counts," else the checklist could never finish there.
- The auto-backup effect no longer attempts a **gesture-less `a.click()` download** when no folder exists (browsers block non-gesture downloads / it dumps a surprise file). It writes silently only when a real folder is present, and records `beacon_last_backup` **only when the write lands off-device** — so the staleness clock no longer lies.

**Still honors CC30:** never auto-writes plaintext PII. Trial protection is **prompt + manual backup**, not silent plaintext auto-backup.

## Gap C (surfaced, open by design)

B-1's trip threshold — "% of active users with a folder backup configured **< 70%**" — is **structurally unmeasurable** in a local-first / no-phone-home app. We'd only learn the threshold was crossed via the OR-clause ("≥1 missing data report"), i.e. the catastrophic way. Not "fixed" (phone-home conflicts with the privacy posture); logged in the risk register as an honest limitation.

## Lessons worth keeping

- **Gate trust-spine features on "has data to lose," not on "has paid."** The auto-backup was implicitly gated on a license key, which silently disabled the product's core durability promise for exactly the cohort with the least invested and the most to lose (a trial counselor's first weeks). Always check what a protective feature does for the *trial/unactivated* state, not just the happy paid path.
- **A flag that's written but never read is a half-shipped feature that looks done.** CC30 designed `beacon_backup_needs_license` "for the UI to prompt"; only the write half landed. Grep the *read* side of any status flag before trusting it.
- **Verify the cohort, not just the code path.** The bug was invisible until the trial/license model was traced (no key during trial). The deterministic gate was correct in isolation; the exposure only appeared when combined with how trials are actually created.
