## 1. Referral-form + CrisisModal 911 wording

## 2. Editable report PDFs (PR #5)

## 3. Crisis addendum at creation (PR #6)

The Crisis Event PDF is **integrity-hashed + attested** ("regenerate → identical hash = unaltered record"), so ad-hoc export-time text would break tamper-evidence. The correct version:
- Optional addendum textarea in the crisis review step → saved on the `crisis_events` record as `counselor_addendum`. Copy says it becomes part of the timestamped record — *"add it now rather than editing later."*
- **Included in the integrity hash, but only when present**, so events without an addendum hash **byte-identically** to before (existing attestations unaffected).
- Rendered as an "Additional Information" PDF section before the signature; signature placement now overflows to a new page instead of clamping.

## 4. Crisis "add-later" surface (PR #7)

Crisis events were write-once in `CrisisModal` with **no way to revisit them**. Added the missing viewer:
- `StudentDetailPage` loads `crisis_events` for the student; a **"Crisis" tab appears only when the student has any** (confidential records — no empty tab in the common case).
- `CrisisRecordsSection` (new) renders a per-event card: trigger/dates/status, the creation note, and every later addendum with its timestamp. **"+ Add addendum"** appends `{ text, added_at }` to the event's `addenda` array via `db.update`; **"Regenerate PDF"** re-runs `generateCrisisPdf`.
- **Tamper-evidence — append-only:** addenda are additions, never edits; the original answers are untouched. The hash covers the `addenda` array (only when present → addendum-less events unchanged); each addendum renders under its own dated heading. A regenerated PDF is a **new, separately attested version** — each version's hash + generated-at stand on their own.
- **No schema/store change** — IndexedDB is schemaless per-record; `addenda` is just a new field. Confirmed `localDb.update` **merges** (`{...existing, ...changes}`), not replaces — the one failure mode that could have wiped a record.

## Lessons worth keeping

- **"All the PDFs" is not one feature — it's two.** Report/summary PDFs can bake in free text safely; integrity-hashed/attested PDFs (crisis, SB-179, parent-contact) cannot — a note must be **persisted to the record and included in the hash**, or "regenerate → same hash" is a check you can't cash. Split the ask along the integrity boundary before building.
- **For tamper-evident records, use append-only addenda, never edits.** Adding info later must be a timestamped addition that the hash then covers (a new attested version), so the original is provably untouched. Editing the original silently would undermine the whole admissibility story.
- **Conditional hash keys preserve backward-compat.** Including a new field in the hash payload **only when present** means existing records hash byte-identically — old attestations stay valid, new content is still covered.
- **When adding shared state, confirm which component's scope you're in.** A multi-component file (`GroupDetailPage` + `LogSessionModal`) made it easy to declare `useState` in the wrong function; lint's "assigned but never used" + "not defined" pair is the tell.
