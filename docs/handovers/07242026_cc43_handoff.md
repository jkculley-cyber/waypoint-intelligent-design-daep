## 3. Adversarial self-audit → graded B+ → fixed to A−

Ran an audit of my own diff, instructed to find bugs rather than praise. It found 10 issues; 6 mattered:

- **CRITICAL** — year-end *Promote & Archive* drove the promotion ladder off the **unsaved** Settings selector. Toggling a band without saving, then running the transition, could graduate the wrong cohort and skip everyone else.
- **HIGH** — the transition dialog over-promised: it counted every student as "will be promoted," then silently skipped out-of-band/blank grades. Now reports promote/graduate/**unchanged** honestly.
- **HIGH** — IndexedDB v7→v8 had no `onblocked` handler. A second open tab left the memoized promise unsettled and **hung the app forever** on a blank screen. **I had explicitly dismissed this in a commit message as "a dev-only artifact, not a code defect." It was a code defect.** That self-deception is the most important thing the audit caught.
- Band changes never brought in the new band's lessons; the `Social/Emotional` domain fix never reached existing users; CCMR records became unreachable after a band change; out-of-band students rendered with a blank grade that invited overwriting; unknown grade tokens vanished from every filter.

**Testing (the real gap):** my verification had been "states, not the moves between them." Closed that structurally — `npm run test:bands`, **1,914 assertions**, 14 counselor states (incl. legacy no-`grade_band`, null, garbage, inverted ranges) × every helper, plus **all 196 state→state transitions**. Plus browser-verified transitions: legacy-user upgrade, band change (lesson top-up 12→13, legacy tag repair 1→0, idempotent re-save), CCMR orphan handling, multi-tab DB yield, K-12 sample spread.

## 4. INCIDENT — the store drip dunned a paying customer ([#29](https://github.com/jkculley-cyber/waypoint-intelligent-design-daep/pull/29), MERGED)

Kim corrected me: **Nicole is already a paying Beacon customer**; the "lead" was an artifact of her demoing the store *during her district presentation*.

That turned a routine lead check into a real incident. Confirmed from the ops drip log:

```
"nhill@magnoliaisd.org|beacon": "2026-07-19T14:50:09.301Z"
```

She holds an **active Beacon license** (expires 2026-08-15) and was sent a *"we haven't seen your payment"* email on 2026-07-19 — three days before this session.

**Root cause:** `computeStoreNudges` only checked the *lead's* status, never whether the registrant was already a customer. So *any* existing customer who walked the registration flow got dunned two days later. Latent since the drip shipped in CC41; Nicole was simply the first to trip it, in the worst possible setting.

**Fixed:** active-license holders are skipped (conservatively, across all products), and the run **fails rather than sends** if the license lookup errors. 23 regression tests including a replay of this exact incident. Lead #48 set to `closed`. Containment was already complete (one-time nudge + sent-marker).

## 5. Live revenue leak — paid PDFs publicly downloadable ([#30](https://github.com/jkculley-cyber/waypoint-intelligent-design-daep/pull/30), MERGED)

All four of Melissa's teacher bundles ($7/$12/$12/$9.50) were downloadable with no payment check — verified live: `application/pdf`, 4.5–11.5 MB.

The first instinct (license-gate via `/activate`) was **wrong**, and building it surfaced why: **no bundle license keys have ever been issued**, so gating would hide the files behind credentials nobody has. And `/activate` is a **client-side** gate — it reveals a link to a file that stays public. Nothing linked the PDFs either; they were orphaned artifacts, not a delivery path.

**Fixed** by `git mv` to `products/teacher-bundles/` (outside the Pages build root), with a README documenting the manual fulfillment path.

## 7. Toolkit download ([#31](https://github.com/jkculley-cyber/waypoint-intelligent-design-daep/pull/31), MERGED)

I had reported this as "returns 0 bytes / broken." **That was my error** — `curl` without `-L`, so I measured the 308 redirect instead of the file. It serves 1,690,528 bytes fine.

The real (smaller) bug: Cloudflare strips `.html`, no `Content-Disposition` is sent, and the link used a bare `download` attribute — so the file saved with **no extension** and wouldn't open on double-click, which is exactly what the page instructs. Fixed with an explicit filename.

## 8. X-2 monitoring — scoped ([#32](https://github.com/jkculley-cyber/waypoint-intelligent-design-daep/pull/32), MERGED) + built (**PR BLOCKED**)

Scope and build both rest on one fact: **a bare 200 proves nothing.** CC31 returned 200 serving the *Waypoint app* at the marketing domain; B-2 returned 200 serving HTML from `/api/*`; every unknown path returns the same 136,780-byte fallback. **A status-only monitor catches none of them.**

Built `scripts/uptime-check.mjs` — 14 checks asserting on **content**, including a check that the homepage does *not* contain `"Waypoint DAEP"` (the exact CC31 signature) and leak-guards that the bundle PDFs stay gone. Read-only, no auth (a probe that needs a credential can die of its own stale secret — itself the X-2 failure mode), standalone (does **not** call a Pages Function, unlike the drip cron: if the site is down that Function is down too), and transient-tolerant (re-test after 20s, second strike only, no persisted state). Alerting is zero-config via workflow failure; `RESEND_API_KEY` upgrades it.

**28 offline tests** replay the CC31 and B-2 payloads and prove both are caught. **First live run: 12/14 — and it immediately caught a real problem** (the two cached bundle PDFs).

**X-2 deliberately stays Active.** DoD requires 7 unattended days with zero false alarms *and* the alert path proven with a deliberately broken assertion. Building a monitor isn't the same as trusting one.
